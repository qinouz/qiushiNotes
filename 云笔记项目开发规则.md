# 云笔记项目开发规则

## 一、项目定位

本项目不是一开始完整复刻有道云笔记，而是先做一款稳定、实用、可长期使用的私人笔记系统。

核心定位：

- Windows 桌面端优先
- 本地优先
- 离线可用
- 数据自己掌控
- 服务器负责同步、备份和后续知识库
- 所有数据必须支持完整导出

第一版的目标是做出一个每天能用的本地笔记软件，而不是一次性完成多端、协作、AI、模板市场等复杂能力。

## 二、核心原则

1. 本地保存不能依赖服务器。
2. 服务器宕机、停电或断网时，用户仍然可以正常打开、编辑、新建和搜索笔记。
3. 所有操作先写入本地 SQLite，再由后台同步任务异步上传。
4. 删除使用软删除，不直接物理删除。
5. 同步冲突不能直接覆盖，必须保留冲突副本。
6. 图片和附件先保存到本地目录，再异步上传服务器。
7. 数据库、附件、导出文件必须能被用户完整备份和迁移。
8. 第一阶段不做复杂云能力，先保证单机稳定。

## 三、第一版功能范围

第一版必须包含：

- 新建、编辑、删除笔记
- 自动保存
- 多级笔记本目录
- 富文本编辑
- 图片粘贴
- 附件本地保存
- 全文搜索
- 回收站
- 本地备份
- 数据导入和导出

第一版可以暂缓：

- 手机 App
- 多人协作
- 复杂权限系统
- AI 问答
- 模板市场
- 一开始就做服务器同步
- 完整复刻有道云笔记所有功能

## 四、推荐技术栈

桌面端：

- Electron
- Vue 3
- TypeScript
- SQLite
- TipTap 富文本编辑器
- Vite

本地架构：

```text
UI 界面层
  ↓
业务服务层
  ↓
SQLite 数据库层
  ↓
本地附件文件层
```

服务器端后续可选：

- Ubuntu Server
- Docker Compose
- Node.js / NestJS
- PostgreSQL
- Redis
- MinIO 或本地对象存储目录

## 五、Electron 版本规则

本项目不追最新 Electron 主线，优先选择仍在维护、但相对稳定的版本。

第一版推荐锁定：

```json
{
  "devDependencies": {
    "electron": "41.7.2"
  }
}
```

规则：

- 不使用 `^41.7.2`，避免开发过程中自动升级 Electron。
- 不为了追新使用刚发布的大版本。
- Electron 升级要单独测试窗口、菜单、SQLite、打包和自动更新。
- 项目稳定前，不频繁升级 Electron 主版本。

## 六、跨平台兼容规则

项目第一阶段优先保证 Windows 可用，但代码从一开始按 Windows / macOS 可兼容方式编写。

必须遵守：

- 不写死 Windows 路径。
- 不直接拼接 `C:\xxx` 这类路径。
- 文件路径统一使用 `path.join(...)`。
- 应用数据目录使用 `app.getPath('userData')`。
- 判断平台使用 `process.platform`。
- 快捷键使用 `CommandOrControl`，不要只写 `Ctrl`。
- macOS 菜单、窗口关闭行为后续单独适配。

示例：

```ts
const dataDir = app.getPath('userData')
const dbPath = path.join(dataDir, 'notes.db')
```

## 七、本地数据规则

本地数据目录建议包含：

```text
userData/
  notes.db
  attachments/
  backups/
  exports/
```

核心表建议：

```text
notes
notebooks
tags
note_tags
attachments
sync_queue
settings
```

笔记表必须提前保留同步相关字段：

```text
id
title
content
notebook_id
created_at
updated_at
deleted_at
version
sync_status
```

即使第一版暂时不做服务器同步，也要保留这些字段，避免后续大改数据库结构。

## 八、开发顺序

在开发任何功能模块之前，必须先编写或更新对应的设计文档。

设计文档统一放在：

```text
docs/designs/zh-CN/
```

每个功能模块对应一份设计文档，例如：

```text
docs/designs/zh-CN/local-database.md
docs/designs/zh-CN/notebook-tree.md
docs/designs/zh-CN/rich-text-editor.md
docs/designs/zh-CN/attachment-storage.md
docs/designs/zh-CN/full-text-search.md
docs/designs/zh-CN/sync-engine.md
```

设计文档至少说明：

- 功能要解决的问题
- 目标和非目标
- 用户操作流程
- UI 行为
- 数据模型
- 本地存储方式
- IPC 或 API 边界
- 未来同步影响
- 异常处理和边界情况
- 隐私与数据安全
- 测试和验证计划
- 暂未决定的问题

如果编码过程中发现设计不完整，必须先更新设计文档，再继续实现。

设计文档默认使用中文。英文文档可以作为辅助参考，但中文文档是以后维护项目时的主要入口。

代码中应适当增加注释，尤其解释：

- 为什么这样划分模块边界
- 为什么要保护本地优先
- 为什么同步还没实现也要保留同步字段
- 数据模型和迁移的关键取舍
- 容易被以后维护时误删或误改的边界情况

注释不要只复述代码做了什么，而要说明为什么这么做。

### 第一阶段：本地桌面原型

目标：

- 应用能启动
- SQLite 能创建
- 能新建笔记
- 能编辑笔记
- 能自动保存
- 关闭后重新打开，数据仍然存在

### 第二阶段：本地核心功能

目标：

- 多级笔记本
- 回收站
- 搜索
- 标签
- 收藏
- 置顶
- 本地备份

### 第三阶段：富文本和附件

目标：

- TipTap 编辑器稳定可用
- 支持标题、加粗、列表、引用、代码块
- 支持图片粘贴
- 支持附件保存和引用
- 支持 Markdown / HTML / JSON 导出

### 第四阶段：服务器同步

目标：

- 设计同步 API
- 实现增量同步
- 实现冲突副本
- 实现附件上传
- 实现服务器备份
- 实现新设备恢复

### 第五阶段：知识库与 AI

目标：

- 笔记分段
- 建立向量索引
- 检索相关笔记
- 调用云端大模型
- 返回答案并引用来源
- 敏感笔记可禁止发送到云端模型

### 第六阶段：移动端

目标：

- Windows 版稳定后再考虑手机 App
- 手机端同样坚持本地优先和离线可用

## 九、同步规则

同步不是第一版目标，但数据模型和业务操作必须为同步预留空间。

同步基本规则：

1. 每条笔记使用 UUID。
2. 本地每次修改都增加版本或更新时间。
3. 本地修改写入 `sync_queue`。
4. 服务器不可用时，修改留在待同步队列。
5. 服务器恢复后自动补传。
6. 服务端不直接覆盖客户端数据。
7. 出现冲突时创建冲突副本。
8. 附件与笔记内容分开同步。

## 十、产品体验规则

界面目标不是炫技，而是安静、稳定、适合长期写作和查找。

优先级：

1. 打开速度
2. 编辑稳定性
3. 自动保存可靠性
4. 搜索速度
5. 数据安全
6. 同步准确性
7. 外观细节

界面结构优先采用：

```text
左侧：笔记本目录
中间：笔记列表
右侧：编辑器
```

避免第一版加入过多视觉装饰、复杂动画和非核心功能。

## 十一、与有道云笔记的关系

有道云笔记可以作为体验参考，但不是完整复刻目标。

优先参考：

- 多端云笔记的基础信息结构
- 笔记本目录
- 笔记列表
- 富文本写作体验
- 图片和附件管理
- 搜索
- 回收站

不优先复刻：

- 会员体系
- 协作体系
- 复杂模板
- 扫描/OCR
- 网页收藏插件
- 商业化功能

本项目的优势应当是：

- 数据自己掌控
- 本地离线可用
- SQLite 可备份
- 附件可直接迁移
- 服务器只是增强，不是依赖
- 后续可接入自己的私人知识库

## 十二、社区项目参考原则

开发前需要参考成熟开源项目，但不直接 fork 成熟项目作为第一版基础。

主要参考对象：

- Joplin：参考桌面端、同步、附件和跨端经验
- Trilium / TriliumNext：参考 SQLite、层级笔记和个人知识库
- Logseq：参考隐私优先、用户掌控、长期知识组织
- AFFiNE：参考 local-first、自托管和未来工作空间方向

本项目的原则是：

- 自研核心数据模型
- 自研本地 SQLite 存储
- 自研同步协议
- 自研附件存储规则
- 借鉴成熟项目的经验，不继承它们的复杂度
