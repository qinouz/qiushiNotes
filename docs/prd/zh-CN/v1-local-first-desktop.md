## 1. 背景

秋实笔记是一款私人云笔记应用，但第一版不应该从“云”开始，而应该从一个稳定、可靠、可离线使用的 Windows 桌面笔记应用开始。

用户已经明确了产品方向：

- 本地数据是事实来源。
- 服务器后续只负责同步、备份、设备恢复和知识库能力。
- 所有用户数据必须可导出、可迁移。
- 不把第一版做成有道云笔记的完整克隆。

这份 PRD 的目的，是在模块设计文档和实现代码之前，先统一第一版产品边界，避免后续直接进入“写某个功能”的局部视角，导致整体信息架构、验收标准和取舍不清楚。

## 2. 参考对象与吸收原则

### 2.1 有道云笔记

参考来源：

- 有道云笔记帮助中心：https://note.youdao.com/help-center/
- 有道云笔记移动端介绍：https://note.youdao.com/WAP/intro/
- 用户提供的有道云笔记桌面端截图

有道云笔记提供了成熟笔记产品的工作流参考：

- 多端资料库。
- 多级目录和笔记组织。
- 搜索、收藏、标签、待办、分享。
- 文字、图片、附件、扫描、OCR、Markdown 等多种记录方式。
- 网页剪藏和外部内容收藏。
- 云端同步、备份、分享和 AI 能力。

第一版只吸收这些“基础工作流和信息架构”：

- 窄功能栏 + 笔记本树 + 列表/编辑区的桌面工作区布局。
- 搜索、新建、全部笔记、笔记本树、收藏、标签、待办、回收站这些入口的长期位置。
- 快速记录、稳定编辑、可靠自动保存的核心体验。

第一版不吸收这些高级能力：

- AI 摘要、智能问答、模板市场、多人协作。
- OCR、扫描、语音、手写。
- 网页剪藏插件。
- 云端实时同步和分享。
- Office/PDF 在线编辑。

### 2.2 Joplin

参考来源：

- Joplin 官方帮助：https://joplinapp.org/help/

Joplin 的关键启发：

- Offline first：所有数据在本机可访问，断网也能使用。
- 支持笔记本、标签、全文搜索、附件、导入导出。
- 支持从其他笔记软件迁移数据。

秋实笔记 V1 应吸收“离线优先”和“数据可迁移”的产品底线，但不照搬插件生态、多平台同步和 Markdown-only 的编辑模式。

### 2.3 Trilium / TriliumNext

参考来源：

- TriliumNext GitHub：https://github.com/TriliumNext/trilium
- Trilium Notes 官网：https://triliumnotes.org/

Trilium 的关键启发：

- 任意深度层级结构适合长期知识库。
- 富文本编辑、图片、表格、代码等能力适合个人知识沉淀。
- 个人知识库更重视组织结构和长期可维护性。

秋实笔记 V1 应支持多级笔记本，但不引入“单条笔记出现在多个位置”“复杂关系图”“个人 wiki 全套能力”等高级结构。

## 3. 产品定位

秋实笔记 V1 是一款本地优先的个人桌面笔记应用。

它的第一版目标不是“功能最多”，而是：

- 能放心写。
- 能放心找。
- 能放心删。
- 能放心备份和导出。
- 断网、服务器未完成、家庭服务器宕机时仍然可用。

一句话定位：

> 面向个人长期使用的本地优先桌面笔记，把可靠编辑、清晰组织、可恢复和可迁移放在第一位。

## 4. 目标用户

### 4.1 核心用户

第一版默认服务一个核心用户：项目作者本人。

用户特征：

- 主要使用 Windows 桌面电脑。
- 有大量日常记录、项目资料、开发笔记、链接、片段、截图和附件。
- 希望数据在自己手里，而不是完全依赖第三方云。
- 能接受第一版没有移动端和协作，但不能接受数据丢失、打不开、导不出。
- 需要未来能同步到家庭服务器，但当前先要离线稳定。

### 4.2 后续可能用户

- 喜欢本地优先软件的个人用户。
- 需要自建同步和备份的开发者。
- 想从有道云笔记、Joplin、Obsidian、印象笔记等工具迁移部分数据的用户。

这些不是 V1 的直接优化目标，但 V1 的数据模型和导出能力要给他们留下空间。

## 5. 产品目标

### 5.1 V1 必须达成

1. 用户可以在 Windows 桌面端创建、编辑、删除和恢复笔记。
2. 所有笔记内容先保存到本地 SQLite。
3. 断网时完整可用，不依赖服务器。
4. 支持多级笔记本，用于整理资料。
5. 支持富文本编辑，至少包含常用段落、标题、列表、引用、代码块、加粗、斜体、链接、图片。
6. 支持图片粘贴和本地附件保存。
7. 支持全文搜索。
8. 支持回收站和恢复。
9. 支持本地备份。
10. 支持数据导入和导出。
11. 数据结构为未来同步保留字段。
12. 所有用户数据可迁移，不被锁死在应用里。

### 5.2 V1 成功标准

V1 完成时，用户应该可以：

- 打开应用后在 2 秒左右看到主界面和最近笔记列表，普通开发机上不出现明显白屏等待。
- 新建一条笔记，输入标题和正文，关闭应用后重新打开仍然存在。
- 粘贴一张图片，图片文件保存到本地附件目录，正文中能再次显示。
- 创建多级笔记本，把笔记放到指定笔记本。
- 搜索标题或正文关键词，并快速定位到目标笔记。
- 删除笔记后在回收站看到它，并恢复到原笔记本。
- 执行一次本地备份，并能从备份中找回用户数据。
- 导出全部或部分笔记到开放格式。
- 在没有网络、没有服务器的情况下完成上述所有操作。

## 6. 非目标

V1 不做：

- 移动端。
- Web 端。
- 多人协作。
- 复杂权限。
- 云端实时同步。
- AI 问答、摘要、写作辅助。
- 模板市场。
- 网页剪藏浏览器插件。
- OCR、扫描、语音、手写。
- Office/PDF 在线编辑。
- 端到端加密。
- 插件系统。
- 完整复刻有道云笔记。

这些能力可以在 V1 稳定后进入 V2/V3 规划。

## 7. 产品原则

### 7.1 本地优先

- 创建、编辑、删除、搜索、回收站、备份、导入导出必须本地完成。
- 服务器不是本地操作的前置条件。
- 未来同步失败不能影响本地编辑。

### 7.2 数据安全优先

- 默认软删除，不物理删除用户数据。
- 批量操作、导入、迁移、清理附件前要有可恢复策略。
- 发生错误时优先保留旧数据。
- 不为了界面简洁隐藏数据风险。

### 7.3 可迁移

- 用户必须能导出自己的笔记和附件。
- 导出格式优先选择通用、可读、可被其他工具处理的格式。
- 本地附件目录和元数据要能对应起来。

### 7.4 克制第一版

- 第一版不做花哨功能。
- 不为了“像某个成熟产品”而加入不稳定模块。
- 每个功能必须服务“放心写、放心找、放心恢复、放心迁移”。

## 8. 第一版信息架构

参考截图和现有项目方向，V1 采用桌面笔记常见工作区：

```text
窄功能栏
  -> 笔记工作区侧栏
      -> 笔记列表
          -> 编辑器
```

### 8.1 窄功能栏

长期入口：

- 最新
- 笔记
- 收藏
- 标签
- 待办
- 回收站
- 设置

V1 实现状态：

| 入口 | V1 状态 | 说明 |
| --- | --- | --- |
| 最新 | P1 | 显示最近编辑笔记，可先并入笔记列表排序 |
| 笔记 | P0 | 默认主入口 |
| 收藏 | P1 | 支持收藏字段和列表入口 |
| 标签 | P1 | 支持标签基础管理和过滤 |
| 待办 | P2 | 先不做独立待办系统 |
| 回收站 | P0 | 必须支持恢复 |
| 设置 | P1 | 数据目录、备份、导入导出入口 |

### 8.2 笔记工作区侧栏

包含：

- 搜索框。
- 新建按钮。
- 全部笔记。
- 笔记本树。
- 回收站入口。

笔记本树必须支持多级展开折叠。V1 可以先做简单排序和右键菜单，拖拽排序后置。

### 8.3 笔记列表

列表显示当前范围内的笔记：

- 全部笔记。
- 某个笔记本及其子笔记本。
- 搜索结果。
- 收藏。
- 标签。
- 回收站。

每条笔记至少显示：

- 标题。
- 正文摘要。
- 更新时间。
- 是否有附件或图片。
- 是否收藏。

### 8.4 编辑器

编辑器区域包含：

- 面包屑：全部笔记 / 笔记本路径 / 当前笔记。
- 标题输入。
- 富文本工具栏。
- 正文编辑区域。
- 保存状态。
- 更多操作菜单。

V1 不做复杂排版装饰，优先让编辑可靠、保存明确、内容可导出。

## 10. 核心用户流程

### 10.1 首次启动

1. 用户打开应用。
2. 应用创建本地数据目录：

   ```text
   userData/
     notes.db
     attachments/
     backups/
     exports/
   ```

3. 应用执行数据库迁移。
4. 应用创建默认笔记本。
5. 主界面显示空状态或默认欢迎笔记。

验收：

- 没有网络也能完成首次启动。
- 如果目录创建失败，显示可理解错误，不静默重建或删除旧数据。

### 10.2 创建和编辑笔记

1. 用户点击“新建”。
2. 如果当前选中某个笔记本，新笔记进入该笔记本；否则进入默认笔记本。
3. 用户输入标题和正文。
4. 应用防抖自动保存。
5. 保存状态显示“保存中 / 已保存 / 保存失败”。

验收：

- 关闭应用再打开，内容仍在。
- 自动保存不会让列表排序频繁跳动。
- 保存失败时不丢失编辑器里的未保存内容。

### 10.3 整理笔记本

1. 用户创建笔记本。
2. 用户创建子笔记本。
3. 用户把笔记移动到指定笔记本。
4. 用户重命名笔记本。
5. 用户删除空笔记本或带内容笔记本。

验收：

- 多级结构可以展开折叠。
- 删除带内容笔记本前必须提示影响。
- 删除笔记本默认进入软删除流程，不能直接物理删除其笔记。

### 10.4 粘贴图片

1. 用户在编辑器中粘贴截图或图片。
2. 应用在 main 进程保存图片到附件目录。
3. 数据库记录附件元数据。
4. 编辑器插入图片引用。
5. 重新打开笔记时图片可显示。

验收：

- 图片文件不保存在仓库。
- 图片路径使用相对路径，便于备份和迁移。
- 图片保存失败时不插入损坏引用。

### 10.5 添加附件

1. 用户通过编辑器或菜单选择文件。
2. 应用复制文件到附件目录。
3. 数据库保存附件记录。
4. 笔记正文中出现附件卡片或链接。

验收：

- 文件名冲突不会覆盖已有附件。
- 大文件有大小限制和错误提示。
- 附件被引用时不能被清理任务误删。

### 10.6 搜索笔记

1. 用户在搜索框输入关键词。
2. 应用搜索标题和正文。
3. 列表显示结果，关键词可高亮。
4. 用户点击结果进入笔记。

验收：

- 搜索不依赖服务器。
- 中文、英文、数字可以搜索。
- 搜索结果不包含已删除笔记，除非用户正在回收站中搜索。

### 10.7 删除和恢复

1. 用户删除笔记。
2. 笔记从普通列表消失，进入回收站。
3. 用户进入回收站查看。
4. 用户恢复笔记。
5. 笔记回到原笔记本；如果原笔记本已删除，则进入默认笔记本并提示。

验收：

- 删除只写 `deleted_at`，不物理删除正文和附件。
- 恢复后笔记内容、图片、附件仍可用。
- 彻底删除必须二次确认，并优先在 V1 后期再开放。

### 10.8 本地备份

1. 用户在设置中点击“立即备份”。
2. 应用生成备份包。
3. 备份包包含 SQLite、附件、必要元数据。
4. 用户可以打开备份目录。

验收：

- 备份过程中不阻塞正常编辑太久。
- 备份失败有明确原因。
- 备份文件不覆盖旧备份。

### 10.9 导出

1. 用户选择导出范围：全部、当前笔记本、当前笔记。
2. 用户选择格式：Markdown、HTML、JSON 包。
3. 应用导出正文和附件。
4. 导出完成后可打开导出目录。

验收：

- 导出的 Markdown/HTML 在普通编辑器或浏览器中可读。
- JSON 包可用于秋实笔记未来重新导入。
- 附件链接能对应导出的附件文件。

### 10.10 导入

1. 用户选择导入来源。
2. 应用预检文件格式和大小。
3. 应用创建导入前备份。
4. 应用导入笔记、笔记本、附件。
5. 导入结果显示成功数、失败数、失败原因。

V1 首选导入格式：

- 秋实笔记 JSON 包。
- Markdown 文件或目录。

后续再考虑：

- Joplin 导出。
- 有道云笔记导出格式。
- Evernote ENEX。

验收：

- 导入失败不能破坏已有数据。
- 重复导入要有去重或明确的重复处理策略。

## 11. 详细功能需求

### 11.1 本地数据目录和启动

需求：

- 应用使用 Electron `app.getPath('userData')` 获取数据目录。
- 内部目录名使用 `QiushiNotes`。
- 启动时确保 `notes.db`、`attachments/`、`backups/`、`exports/` 存在。
- 数据库迁移幂等执行。

验收标准：

- 删除空数据目录后启动，应用能重新初始化。
- 已存在数据库时启动不会覆盖。
- 迁移失败时停止后续写入，并提示用户备份/恢复。

### 11.2 笔记模型

字段要求：

```text
id
title
content
content_format
notebook_id
created_at
updated_at
deleted_at
version
sync_status
is_favorite
is_pinned
```

需求：

- `id` 使用 UUID。
- `content_format` V1 支持 `tiptap-json`，导入纯文本或 Markdown 时可转换。
- `deleted_at` 为空表示正常笔记。
- 每次用户编辑增加版本号。
- `sync_status` 即使同步未实现也要维护。

验收标准：

- 新建、更新、删除后字段符合预期。
- 旧版本纯文本笔记能迁移或兼容显示。

### 11.3 笔记本模型

字段要求：

```text
id
parent_id
name
sort_order
created_at
updated_at
deleted_at
version
sync_status
```

需求：

- 支持任意层级，但 UI 上建议控制缩进和性能。
- 同级笔记本名称不强制唯一，但重名时 UI 要能区分。
- 删除父笔记本时必须处理子笔记本和笔记。

验收标准：

- 能创建至少 5 层嵌套笔记本。
- 重启后展开状态可后续保存，V1 可以先不保存。
- 移动笔记本时不能形成循环父子关系。

### 11.4 富文本编辑器

需求：

- 使用 TipTap。
- 支持标题、段落、列表、任务列表、引用、代码块、分割线、链接、图片、加粗、斜体、删除线。
- 保存为结构化 JSON。
- 导出时转换为 Markdown 或 HTML。

验收标准：

- 基础格式重启后不丢失。
- 粘贴普通文本、富文本、链接时行为稳定。
- 编辑器内容为空时不报错。

### 11.5 自动保存

需求：

- 用户输入后防抖保存。
- 标题和正文都参与自动保存。
- 保存状态明确。
- 离开当前笔记前尽量 flush 未保存内容。

验收标准：

- 快速连续输入不会每个字符都写数据库。
- 保存失败时用户能看到，并且编辑器内容仍保留在内存中。
- 切换笔记不应覆盖另一条笔记。

### 11.6 图片和附件

需求：

- 图片粘贴保存到本地附件目录。
- 普通附件复制到附件目录。
- 附件元数据进入 SQLite。
- 正文引用使用附件 ID 或相对路径，不使用绝对路径。

验收标准：

- 迁移整个 userData 目录后图片仍可显示。
- 删除笔记不立即删除附件。
- 恢复笔记后附件仍可用。

### 11.7 全文搜索

需求：

- 支持标题和正文搜索。
- 优先使用 SQLite FTS。
- 支持搜索结果排序。
- 已删除笔记默认不参与普通搜索。

验收标准：

- 搜中文关键词能找到包含该词的笔记。
- 搜标题比正文命中优先级更高。
- 1000 条普通文本笔记下搜索仍然可接受。

### 11.8 回收站

需求：

- 删除笔记进入回收站。
- 回收站展示删除时间、原笔记本。
- 支持恢复。
- 支持彻底删除，但需要二次确认。若实现风险较高，可放到 V1 后半段。

验收标准：

- 删除后普通列表不可见。
- 回收站可见。
- 恢复后普通列表可见。
- 附件不丢。

### 11.9 备份

需求：

- 支持手动备份。
- 支持关键操作前安全备份，例如导入、迁移。
- 备份包至少包含 `notes.db` 和 `attachments/`。
- 备份文件命名包含时间。

验收标准：

- 备份包可以被人工识别。
- 连续备份不会覆盖旧文件。
- 备份失败不影响当前数据库。

### 11.10 导入导出

导出需求：

- Markdown 导出：适合人类阅读和迁移。
- HTML 导出：保留富文本展示。
- JSON 包导出：适合秋实笔记完整恢复。

导入需求：

- JSON 包导入。
- Markdown 文件/目录导入。
- 导入前预检。
- 导入前创建备份。

验收标准：

- 至少一条包含标题、正文、图片的笔记能完整导出并重新导入。
- 导入失败时已有数据不被破坏。

### 11.11 标签和收藏

V1 优先级为 P1。

需求：

- 笔记可收藏。
- 可按收藏查看。
- 可创建标签。
- 可给笔记绑定多个标签。
- 可按标签过滤笔记。

验收标准：

- 收藏和标签重启后保留。
- 删除笔记后标签关系不出现在普通列表中。

### 11.12 设置

V1 设置页包含：

- 数据目录位置。
- 打开数据目录。
- 打开备份目录。
- 立即备份。
- 导入。
- 导出。
- 应用版本和诊断信息。

验收标准：

- 不暴露危险的直接数据库编辑入口。
- 路径显示和打开行为在 Windows 上可用。

- 未来可以按 hash 或日期分目录。

## 13. IPC / API 边界

标准调用链：

```text
Vue renderer
  -> window.qiushi.*
  -> preload
  -> ipcRenderer.invoke(...)
  -> ipcMain.handle(...)
  -> main service
  -> SQLite / filesystem
```

V1 需要的 preload API 分组：

```ts
window.qiushi.notes
window.qiushi.notebooks
window.qiushi.tags
window.qiushi.attachments
window.qiushi.search
window.qiushi.trash
window.qiushi.backup
window.qiushi.importExport
window.qiushi.settings
```

原则：

- renderer 不直接访问 SQLite。
- renderer 不直接访问本地文件系统。
- main service 负责数据规则。
- IPC 输入必须校验。

## 14. UI 需求

### 14.1 总体风格

- 安静、实用、工作导向。
- 不做营销式首页。
- 不做复杂动画。
- 不用大量装饰卡片。
- 信息密度适中，适合长期写作和整理。

### 14.2 布局

V1 推荐布局：

```text
窄功能栏：固定入口
侧栏：搜索、新建、笔记本树
列表栏：当前范围笔记列表
编辑区：标题、工具栏、正文
```

### 14.3 截图参考提炼

从用户提供的有道云笔记截图中吸收：

- 左侧窄导航保持功能入口稳定。
- 搜索和新建放在高频位置。
- 笔记本树支持层级缩进和选中高亮。
- 编辑区尽量大，减少干扰。
- 面包屑显示当前笔记位置。

不吸收：

- VIP、会员、商业入口。
- AI 摘要按钮作为 V1 核心入口。
- 分享、协作、云功能作为第一版主流程。

### 14.4 空状态

需要为空状态设计明确动作：

- 没有笔记：显示“新建第一条笔记”。
- 没有搜索结果：显示“没有找到相关笔记”。
- 回收站为空：显示“回收站为空”。
- 没有笔记本：自动创建默认笔记本。

## 15. 非功能需求

### 15.1 性能

- 普通数据量下启动尽量在 2 秒左右进入主界面。
- 自动保存不能明显阻塞输入。
- 搜索 1000 条普通笔记应在可接受时间内返回。
- 列表渲染应考虑未来虚拟滚动。

### 15.2 稳定性

- 应用崩溃后，已保存内容不能丢。
- 自动保存失败不应清空编辑器。
- 数据库迁移失败不能破坏旧库。

### 15.3 隐私

- V1 不上传任何笔记内容。
- 错误日志不记录完整笔记正文、附件内容或敏感路径。
- 未来 AI 和同步必须显式设计隐私边界。

### 15.4 可维护性

- 功能实现前先更新设计文档。
- 关键数据规则在 main service 中集中维护。
- 代码注释解释本地优先、同步预留字段和数据安全取舍。

## 16. 错误处理和边界情况

必须覆盖：

- 数据目录无法创建。
- 数据库无法打开。
- 数据库迁移失败。
- 笔记保存失败。
- 附件复制失败。
- 图片粘贴格式不支持。
- 搜索索引损坏。
- 导入文件格式错误。
- 备份目录不可写。
- 原笔记本已删除时恢复笔记。
- 用户快速切换笔记时仍有未保存内容。

处理原则：

- 先保护数据。
- 再提示用户。
- 最后才尝试恢复或重试。

## 17. 安全需求

V1 虽然主要是本地应用，也必须遵守安全边界：

- 不信任 renderer 输入。
- 所有 IPC 输入做类型和范围校验。
- 文件路径必须限制在允许目录内。
- 防止路径穿越。
- 富文本 HTML 渲染必须清洗。
- 不执行导入内容中的脚本。
- 不把用户输入拼接进 SQL。
- 不记录敏感内容到日志。

## 22. 第一版不变承诺

V1 无论后续功能如何调整，都不改变这些承诺：

- 本地编辑不依赖服务器。
- 用户数据先写本地 SQLite。
- 图片和附件先保存本地。
- 删除优先软删除。
- 导入和迁移优先保护旧数据。
- 用户能导出自己的数据。
- 服务器和 AI 不是 V1 的前置条件。
