# 社区成熟项目参考

## 状态

草稿

## 背景

这个项目不应该闭门造车。笔记、知识库、本地优先、同步、导出这些问题，社区里已经有不少成熟项目踩过坑。

我们的目标是：认真参考成熟项目的经验，但不直接继承它们的复杂度、历史包袱和产品方向。

## 参考项目

### Joplin

官方链接：

- https://joplinapp.org/help/dev/spec/architecture/
- https://github.com/laurent22/joplin

值得参考：

- 成熟的跨平台笔记应用。
- 桌面端、同步、附件、标签、搜索、导入导出都比较完整。
- Joplin Server 专门负责多设备同步。

借鉴点：

- 同步应该是独立系统，不能成为本地编辑的前置条件。
- 桌面端即使没有同步，也应该能正常使用。
- 附件资源要和笔记内容分开建模。
- 导入导出从一开始就要认真设计。

暂不借鉴：

- 完整多客户端复杂度。
- Joplin 现有同步协议。
- 待办事项、插件生态等第一版非核心能力。

### Trilium / TriliumNext

官方链接：

- https://github.com/TriliumNext/trilium
- https://triliumnotes.org/

值得参考：

- 面向大型个人知识库。
- 层级笔记是一等能力。
- 使用 SQLite 存笔记、树结构、元数据和配置。
- 一条笔记可以出现在多个位置。

借鉴点：

- SQLite 可以作为长期本地文档存储。
- 笔记本树要为大量长期资料做准备。
- 元数据和关系要显式建模，不要只藏在 UI 状态里。

暂不借鉴：

- 深度个人 wiki 复杂度。
- 第一版不做笔记克隆、多位置挂载。
- 第一版不做大量特殊笔记类型和脚本能力。

### Logseq

官方链接：

- https://github.com/logseq/logseq
- https://github.com/logseq/logseq/blob/master/CODEBASE_OVERVIEW.md

值得参考：

- 隐私优先。
- 强调用户控制和长期可用。
- Markdown / Org 文件是重要数据形态。
- 知识图谱和双链适合作为长期方向。

借鉴点：

- 用户数据所有权本身就是产品能力。
- 导出格式要可读、耐久。
- 双链和知识图谱可以后续考虑，不进入第一版。

暂不借鉴：

- 大纲优先编辑模型。
- 文件即主存储的第一版方案。
- 图谱缓存和知识图谱复杂度。

### AFFiNE

官方链接：

- https://github.com/toeverything/AFFiNE
- https://docs.affine.pro/self-host-affine/

值得参考：

- local-first 工作空间方向。
- 支持自托管。
- 现代块编辑器和画布方向。

借鉴点：

- 本地优先可以作为长期产品身份。
- 自托管是严肃的部署路径。
- 核心笔记稳定后，可以为更丰富的知识库能力留空间。

暂不借鉴：

- Notion / Miro 式复杂工作空间。
- 画布优先编辑。
- 协作优先架构。
- 重型服务端工作区模型。

## 决策

第一版不 fork 成熟开源笔记项目。

我们采用小型自研架构：

- Electron 桌面应用。
- Vue 3 渲染层。
- TypeScript。
- SQLite 作为本地事实来源。
- 附件保存在本地目录。
- 服务器同步后续再作为增强能力加入。
- 功能开发前先写设计文档。

## 理由

直接 fork 成熟项目可以快速得到很多功能，但也会带来：

- 巨大代码库理解成本。
- 数据模型不一定适合我们的本地优先和导出目标。
- 同步协议不一定适合家庭服务器。
- 产品假设很难完全改掉。

自研核心能让我们掌控：

- 本地数据库结构。
- 同步协议。
- 附件存储规则。
- 导出格式。
- 未来私人知识库和 AI 集成方式。

## 架构经验

- 本地编辑必须独立于同步。
- 同步必须异步、可恢复。
- 附件要和笔记分开建模。
- 层级结构是一等能力。
- 导出和备份要早设计。
- 稳定、耐用，比功能数量更重要。

## 待确认问题

- 后续是否支持一条笔记出现在多个笔记本位置？
- Markdown 是否只是导出格式，还是未来支持往返导入？
- 双链是否需要提前留字段？
- 同步模型更接近 Joplin 的 item sync，还是自定义 change-log sync？
