# Security and Hardening

## 何时使用

- 处理用户输入、富文本、HTML、Markdown、搜索关键字。
- 处理图片粘贴、附件、导入、导出、备份、路径。
- 新增 IPC 接口、主进程服务、文件系统访问。
- 未来新增账号、服务器同步、外部 API、Webhook、AI 功能。

## 本项目安全基线

- Renderer 不直接访问 SQLite 或本地文件系统。
- 所有本地数据操作必须经由 preload 暴露的受控 API，再进入 main 进程服务。
- 不信任 renderer 输入，即使它来自本机 UI。
- 不物理删除用户数据，除非用户明确执行不可恢复操作且设计文档说明清楚。
- 不把秘密、令牌、完整路径、敏感内容写入日志。
- 不拼接 SQL；统一使用参数化查询或已有安全封装。
- 文件路径必须基于 `app.getPath('userData')` 和 `path.join(...)`，并防止路径穿越。

## 快速威胁建模

每次处理安全相关功能，先回答：

- 信任边界在哪里？例如 renderer -> preload -> main、导入文件 -> 解析器、附件文件名 -> 文件系统。
- 资产是什么？例如笔记正文、附件、备份、导出文件、未来账号令牌。
- 攻击或误用方式是什么？例如路径穿越、HTML 注入、巨型文件导致卡死、恶意导入包覆盖数据。
- 失败时如何保护用户数据？例如保留原文件、创建备份、导入前预检。

## 必做

- 在边界处校验输入类型、长度、路径、文件大小和允许格式。
- 数据库访问使用参数化语句。
- 富文本或 HTML 渲染必须有明确清洗策略，不把不可信内容直接写入 `innerHTML`。
- 文件写入先落到安全目录，必要时使用临时文件再原子替换。
- 导入、迁移、清理前考虑备份和可恢复性。
- 错误消息对用户友好，对日志不过度泄露。

## 需要先问用户

- 新增外部服务或云同步行为。
- 新增认证、权限、令牌保存方式。
- 新增可执行脚本、插件或自动化执行能力。
- 物理删除用户内容、附件或备份。
- 放宽 Electron 安全配置或 CORS/网络限制。

## 禁止

- 提交密钥、令牌、密码。
- 把认证令牌放进 localStorage。
- 用用户输入拼接 SQL、shell 命令或文件路径。
- 因为开发方便而关闭安全头、隔离、上下文保护。
- 在没有清洗的情况下渲染不可信 HTML。

## 验收清单

- [ ] 信任边界已识别。
- [ ] 输入、路径、文件大小和格式有校验。
- [ ] 不会破坏本地优先和可恢复性。
- [ ] 没有新增秘密泄露或日志泄露。
- [ ] 高风险场景有测试或手动验证。
