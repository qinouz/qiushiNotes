# Doubt-Driven Development

## 何时使用

当一个决定“看起来合理但错了会很贵”时启用。典型场景：

- SQLite 迁移、软删除、恢复、备份、导入导出。
- 附件路径、图片粘贴、文件移动、清理逻辑。
- IPC 边界、权限边界、Electron 安全配置。
- 搜索索引、自动保存、防抖、并发写入。
- 未来同步相关字段、版本号、冲突副本。

机械改名、格式化、小型文案调整不需要启用。

## 核心原则

对关键结论做反向质疑，而不是等代码写完才 review。目标不是拖慢开发，而是在改动还小的时候发现错误方向。

## 流程

1. 写出主张

   用两三句话说清楚当前判断：

   ```text
   主张：这次软删除只更新 deleted_at，不物理删除附件文件。
   重要性：如果附件被提前删除，恢复笔记会造成用户数据丢失。
   ```

2. 抽取可审查单元

   只拿最小必要内容审查：相关函数、设计片段、迁移方案或差异，而不是整仓库。把“契约”写清楚，例如“删除笔记后必须可从回收站恢复”。

3. 反向审查

   用“找问题”的视角检查：

   - 有哪些未说明假设？
   - 哪些边界输入会破坏契约？
   - 是否会违反本地优先？
   - 是否可能导致数据丢失、重复写入或无法恢复？
   - 是否和未来同步字段冲突？

4. 归类发现

   每个发现只能归入一种：

   - 契约不清：先补设计文档或任务说明。
   - 有效且需修复：改设计或代码，并重新检查。
   - 有效但接受的取舍：记录理由。
   - 噪音：说明为什么不是问题。

5. 停止条件

   一轮只剩轻微问题即可停止。最多三轮；三轮后仍有实质问题，应暂停并向用户说明风险。

## 验收清单

- [ ] 高风险主张被明确写出。
- [ ] 审查单元足够小。
- [ ] 审查输入包含契约，不包含自我辩护。
- [ ] 所有发现已归类。
- [ ] 有效问题已经修复或记录为明确取舍。
