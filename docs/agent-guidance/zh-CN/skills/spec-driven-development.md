# Spec-Driven Development

## 何时使用

- 开始一个新功能模块或会触碰多个文件的改动。
- 需求还不清楚，或者“完成”的标准无法直接测试。
- 涉及 SQLite 数据模型、IPC 边界、附件存储、导入导出、备份、同步字段。
- 预计实现超过 30 分钟。

小型 typo、单行修复、无行为变化的格式调整可以跳过完整流程。

## 本项目化规则

本项目已经规定“功能模块开发前先写中文设计文档”，所以这里的 spec 不另起炉灶，优先落在：

```text
docs/designs/zh-CN/<feature-module>.md
```

如果已有对应设计文档，先更新它，再改代码。设计文档应覆盖目标、非目标、用户流程、UI 行为、数据模型、本地存储、IPC/API 边界、同步影响、错误处理、隐私和数据安全、测试计划、开放问题。

## 流程

1. 写出假设

   在动手前列出关键假设，例如：

   - 数据是否必须先写入本地 SQLite。
   - 是否允许物理删除文件或记录。
   - 是否会影响未来同步冲突处理。
   - 是否需要新增依赖、迁移或 IPC 接口。

2. 更新设计文档

   把需求转成可验证的成功标准。不要只写“支持导出”，而要写清楚导出格式、导出位置、失败行为、用户数据是否可迁移。

3. 制定实现计划

   计划应按依赖顺序排列，并说明每一步如何验证。涉及数据库迁移时，必须说明迁移原因、兼容性和回滚/恢复思路。

4. 拆成任务

   每个任务尽量只完成一个行为切片，包含：

   - 要改的文件范围。
   - 验收条件。
   - 验证命令或手动检查步骤。

5. 实现时保持文档同步

   如果编码发现设计不完整，先补设计文档，再补实现。

## 验收清单

- [ ] 中文设计文档存在或已更新。
- [ ] 成功标准具体、可测试。
- [ ] 明确哪些事情必须本地优先完成。
- [ ] 明确是否影响 SQLite、附件目录、IPC、导入导出或未来同步。
- [ ] 明确测试和手动验证方式。
