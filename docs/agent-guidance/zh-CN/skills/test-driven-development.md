# Test-Driven Development

## 何时使用

- 新增或修改核心业务行为。
- 修复 bug，尤其是数据丢失、搜索错误、恢复失败、自动保存异常。
- 改 SQLite schema、迁移、服务层、IPC、导入导出、附件管理。

## 本项目测试优先级

1. 数据安全：软删除、恢复、备份、导入导出、附件引用。
2. 本地优先：离线创建、编辑、搜索不依赖服务器。
3. IPC 边界：renderer 只能通过 preload 暴露的 API 访问本地能力。
4. UI 工作流：笔记创建、选择、编辑、自动保存、笔记本树过滤。
5. 性能边界：大笔记列表、大正文、频繁输入时不阻塞。

## 流程

1. 先读设计文档和现有测试习惯。
2. 为行为写一个会失败的测试，或在缺少测试框架时写出可执行的验证步骤。
3. 确认失败原因和目标 bug/需求一致。
4. 写最小实现让测试通过。
5. 补边界条件测试。
6. 运行相关测试、类型检查和必要的构建。

## Bug 修复规则

修 bug 时优先使用“证明它确实坏了”的测试：

- 测试应在修复前失败。
- 测试名像规格说明一样描述用户可见行为。
- 不测私有实现细节，只测对外契约。

## 测试层级

- 纯函数、数据转换：单元测试。
- SQLite 服务、IPC、文件系统：集成测试。
- 关键用户路径：端到端或手动验证，必要时用浏览器/应用截图确认。

## 验收清单

- [ ] 新行为或 bug 有测试/验证步骤覆盖。
- [ ] 覆盖空值、边界值、错误路径。
- [ ] 数据安全行为至少有一条防回归验证。
- [ ] 测试命令或手动验证结果已记录。
